AOP,不通过修改源代码的方式添加新功能.

AOP使用动态代理实现功能增强.如果有接口,使用JDK动态代理,如果没有接口,使用CGLIB动态代理.前者是创建接口的实现类来增强被代理类功能.后者是通过创建子类来实现功能增强.

连接点:类里面哪些方法可以被增强,这些方法成为连接点.

切入点:实际被增强的方法就是切入点

通知(增强):实际增强的逻辑部分,有五种类型,前置通知,后置通知,环绕通知,异常通知,最终通知.

切面:是一种行为动作,把通知应用到切入点的过程.例如在登录前做日志记录之中前置通知就是切面.

利用AspectJ实现AOP操作:

切入点表达式:表示对类里哪个方法进行增强

语法:execution([权限修饰符][返回类型][类的全类名]方法名称)

例子:对com.company里UserDao的add方法增强:

execution(* com.compony.UserDao.add(..)) //*表示所有修饰符,..表示方法参数,返回类型被省略

例子:对对com.company里UserDao的所有方法增强:

execution(* com.compony.UserDao.*(..))

对com.company里所有类的所有方法增强:

execution(* com.compony.*.*(..))

IOC原理

xml解析,工厂模式,反射

对象之间的相互调用的原始方式是直接在A类中new一个B类对象,这样耦合度太高.改进就是用工厂,在工厂中创建对象.而IOC就是用xml解析得到bean的类名,然后在工厂中根据类名创建对象,进一步降低耦合.IOC的底层就是对象工厂.spring提供两种IOC容器实现方式(两个接口):BeanFactory,ApplicationContext.前者是IOC容器的基本实现,是Spring内部使用的接口,开发人员一般不使用,后者是前者的子接口,有更多强大的子功能,开发人员使用.

大致意思就是ThreadLocal提供了线程内存储变量的能力，这些变量不同之处在于每一个线程读取的变量是对应的互相独立的。通过get和set方法就可以得到当前线程对应的值。

总结如下：

对于某一ThreadLocal来讲，他的索引值i是确定的，在不同线程之间访问时访问的是不同的table数组的同一位置即都为table[i]，只不过这个不同线程之间的table是独立的。

对于同一线程的不同ThreadLocal来讲，这些ThreadLocal实例共享一个table数组，然后每个ThreadLocal实例在table中的索引i是不同的。

理解了set方法，get方法也就清楚明了，无非是通过计算出索引直接从数组对应位置读取即可。

ThreadLocal实现主要涉及Thread，ThreadLocal，ThreadLocalMap这三个类。关于ThreadLocal的实现流程正如上面写的那样，实际代码还有许多细节处理的部分并没有在这里写出来。

ThreadLocal特性

ThreadLocal和Synchronized都是为了解决多线程中相同变量的访问冲突问题，不同的点是

  • Synchronized是通过线程等待，牺牲时间来解决访问冲突
  • ThreadLocal是通过每个线程单独一份存储空间，牺牲空间来解决冲突，并且相比于Synchronized，ThreadLocal具有线程隔离的效果，只有在线程内才能获取到对应的值，线程外则不能访问到想要的值。

正因为ThreadLocal的线程隔离特性，使他的应用场景相对来说更为特殊一些。在android中Looper、ActivityThread以及AMS中都用到了ThreadLocal。当某些数据是以线程为作用域并且不同线程具有不同的数据副本的时候，就可以考虑采用ThreadLocal。

补充

ThreadLocal是一个弱引用,当ThreadLocal变为null后,就会被回收,即ThreadLocalMap中entry中的key没了,但是ThreadLocalMap的生命周期和ThreadLocal并不一样,这是就出现了key没了,value还在的问题,所以要用remove方法来删除对应的value.

ThreadLocal用来解决事务管理问题.在数据库层面进行事务操作是不够的,因为只能保证当前数据库连接的事务.如果对数据库的操作是在不同连接进行的,也就没法使操作在同一事务中进行.所以要用ThreadLocal保存数据库连接,使得相关的数据库操作都在同一个连接中的事务中.

Session:session是用来维护一个客户端和服务器之间关联的一种技术.

  1. 由于HTTP协议是无状态的协议，所以服务端需要记录用户的状态时，就需要用某种机制来识具体的用户，这个机制就是Session.典型的场景比如购物车，当你点击下单按钮时，由于HTTP协议无状态，所以并不知道是哪个用户操作的，所以服务端要为特定的用户创建了特定的Session，用用于标识这个用户，并且跟踪用户，这样才知道购物车里面有几本书。这个Session是保存在服务端的，有一个唯一标识。在服务端保存Session的方法很多，内存、数据库、文件都有。集群的时候也要考虑Session的转移，在大型的网站，一般会有专门的Session服务器集群，用来保存用户会话，这个时候 Session 信息都是放在内存的，使用一些缓存服务比如Memcached之类的来放 Session。

  2. 思考一下服务端如何识别特定的客户？这个时候Cookie就登场了。每次HTTP请求的时候，客户端都会发送相应的Cookie信息到服务端。实际上大多数的应用都是用 Cookie 来实现Session跟踪的，第一次创建Session的时候，服务端会在HTTP协议中告诉客户端，需要在 Cookie 里面记录一个Session ID，以后每次请求把这个会话ID发送到服务器，我就知道你是谁了。有人问，如果客户端的浏览器禁用了 Cookie 怎么办？一般这种情况下，会使用一种叫做URL重写的技术来进行会话跟踪，即每次HTTP交互，URL后面都会被附加上一个诸如 sid=xxxxx 这样的参数，服务端据此来识别用户。

  3. Cookie其实还可以用在一些方便用户的场景下，设想你某次登陆过一个网站，下次登录的时候不想再次输入账号了，怎么办？这个信息可以写到Cookie里面，访问网站的时候，网站页面的脚本可以读取这个信息，就自动帮你把用户名给填了，能够方便一下用户。这也是Cookie名称的由来，给用户的一点甜头。

所以，总结一下：Session是在服务端保存的一个数据结构，用来跟踪用户的状态，这个数据可以保存在集群、数据库、文件中；Cookie是客户端保存用户信息的一种机制，用来记录用户的一些信息，也是实现Session的一种方式。

当客户端第一次发出请求时,没有cookie,服务器创建会话对象session,而服务器每次创建session对象都会创建一个cookie对象,该cookie的key为JSESSION,value就是session的id值.这个特别的cookie发给客户端后,客户端就创建这个cookie对象.以后客户端的每次请求就会把cookie形式的session id发给服务端.然后通过这个传过来的id值去找内存中的session对象.

表单重复提交:

1.表单提交后刷新页面又一次提交,造成数据重复.解决方法是在doPost方法中成功处理数据提交后将页面重定向到新页面,此时刷新的就是新页面,不会重复提交数据.

2.用户正常提交,但是由于网络等原因迟迟未收到服务器响应信息,用户又点击提交.

3.用户提交后进入其他页面,又回退到提交页面,也会重复提交.

后两种情况要用到验证码.

当用户第一次请求登录页面,要给表单随机生成一个验证码字符串,然后把这个验证码存到session中,还要把验证码生成为图片显示在表单中.用户收到登陆页面的响应后开始填写信息和验证码(填的验证码在表单中)并提交.服务端收到信息后先获取session中的验证码并删除,获取表单信息和表单中的验证码,并将这个填写的验证码和session中的比较.相等就允许操作,不相等就不允许操作.那么第一肯定是可以的(只要验证码没写错),第二次就不可以,因为此时session中验证码为空,和用户提交的肯定不一样.

使用验证码需要导入谷歌的kaptcha包.

select @@transaction_isolation;#查看隔离级别

MySQL提供了四种隔离级别:

READ UNCOOMMITED:允许事务读取其他事务未提交的变更,脏读,不可重复读,幻读都会出现

READ COMMITED:允许事务读取已经提交的变更,可避免脏读

REPEATABLE READ:确保事务多次从同一个字段读取出相同的值,在这个事务期间禁止其他事务修改该字段,可以避免脏读和不可重复读(默认级别)

SERIALIZABLE:类似于Java中的加锁,可以避免所有并发问题,但性能较低

set session transaction isolation level serializable;#设置当前会话的隔离级别
